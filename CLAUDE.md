# CLAUDE.md — house rules for this repo

This file is read by Claude Code on every session. Keep it short. Keep it
accurate. When something here becomes false, update it in the same PR that
made it false.

## Project

**Pomodoro CLI (`pomo`)** — a Python 3.10+ command-line Pomodoro timer with
SQLite session logging. See `docs/prd-pomodoro-cli.pdf` for the full PRD.

## Ground truth

- **PRD:** `docs/prd-pomodoro-cli.pdf` (source: `docs/prd-pomodoro-cli.md`)
- **Tasks:** `02-tasks/tasks.json` — the only place where tickets are
  authored. Never hand-create GitHub issues; re-run the sync script instead.
- **Sync mapping:** `sync-report.json` at repo root — maps `T-XXX` → issue
  number. Regenerated by `03-sync/sync_to_github.py`.

## Commands you will actually run

```bash
# Install dev environment
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"

# Test
pytest -q                                  # fast feedback
pytest --cov=pomo --cov-report=term-missing # coverage
pytest -k "test_timer" -v                  # a subset

# Lint & type-check
ruff check .
ruff format --check .
mypy pomo/

# Run the CLI locally
pomo --help
pomo start --work 1 --break 1 --tag smoke  # 2-min smoke test
```

## Test philosophy

- Unit tests are pure Python, no real sleeps, no real filesystem beyond
  `tmp_path`, no network. Use `FakeClock` for anything time-dependent.
- Integration tests that touch the DB use `POMO_DB_PATH=:memory:`.
- CLI tests use Typer's `CliRunner`.
- Target ≥ 90% coverage on `pomo/core/` and `pomo/storage/`. Anything below
  that is a failing CI signal, not a suggestion.

## Style

- `ruff` is the source of truth for formatting and linting. If ruff is happy,
  the code is formatted correctly.
- Type hints on every public function. `Any` requires a comment explaining
  why a real type isn't feasible.
- Docstrings: one line for simple helpers, full triple-quoted block with
  Args/Returns/Raises for anything in a public API.
- No f-string side effects; no mutable default args; no bare `except`.

## Commits

- Conventional commits: `type(scope): subject`
- Types used here: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `infra`
- Include `Refs: T-KEY` in every commit. `Closes: #N` only in the last
  commit of a PR.
- Plain, factual subject lines. No emoji in commit titles. No self-promo
  trailers.

## Branching

- Default branch: `main`. Protected: no direct pushes.
- Feature branches: `feat/T-XXX-short-slug`. One ticket per branch.
- Rebase, don't merge, before opening a PR if `main` has moved.

## Secrets

- **Never** commit `.env`, `*.db`, `.venv/`, local config files, or tokens.
- CI secrets live in GitHub Actions secrets. See `.github/workflows/`.

## Things the PM pipeline owns

You (Claude Code) do not:
- create or edit GitHub labels, milestones, issues, or Project fields
  directly. Those come from the sync script.
- invent new ticket keys. If you think a task is missing, tell the user and
  let them add it to `tasks.json` upstream.

You do:
- implement whatever the current ticket says.
- push back (via issue comments) if a ticket is ambiguous or out of scope.

## When in doubt

Read the ticket again, then the PRD, then ask. In that order.
